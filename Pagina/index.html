Fase 1: Introducción y Modelado

Bienvenidos a su primera gran inmersión en el desarrollo web profesional. Dominar esta arquitectura les abrirá las puertas al mundo laboral real.

1. La Analogía del Restaurante: ¿Qué es MVC?

Antes de tocar una sola línea de código, necesitan entender cómo funciona la arquitectura MVC (Modelo-Vista-Controlador). Imaginen que el sistema de software que vamos a programar es un Restaurante de Alta Cocina:

[Image of MVC (Model-View-Controller) architecture diagram]
  • La Vista (View): Es el Comedor y el Menú. Es todo lo que el cliente (usuario) puede ver, tocar e interactuar (botones, tablas, colores). La Vista no sabe cómo cocinar, solo sabe verse presentable y mostrar las opciones. En nuestro código, serán los archivos HTML.
  • El Controlador (Controller): Es el Mesero. El cliente no entra a la cocina a prepararse su comida. El cliente llama al mesero (Controlador), le hace un pedido (ej. "Quiero ver mis reservas"), el mesero va a la cocina, pide la información, y luego se la sirve de vuelta al cliente en su mesa. Es el coordinador o intermediario perfecto.
  • El Modelo (Model): Es la Cocina, los Ingredientes y las Recetas. Aquí es donde están los datos reales, las reglas matemáticas del negocio y la conexión con la despensa (la base de datos). El Modelo sabe exactamente qué datos conforman a un "Funcionario" o una "Reserva".

2. Entendiendo la Estructura de Carpetas

El patrón MVC divide nuestro proyecto en un esqueleto de carpetas perfectamente organizado. Las piezas más importantes, basadas en nuestra analogía, son:

  • 📂 Models: Tu cocina. Aquí crearemos archivos .cs (Clases de C#) que representarán las "plantillas" de nuestros datos (ej. cómo está compuesta una Reserva).
  • 📂 Views: Tu comedor. Aquí vivirán archivos .cshtml (una mezcla genial de HTML y C#) que serán las pantallas visuales de tu sistema web.
  • 📂 Controllers: Tus meseros. Aquí guardaremos archivos .cs que recibirán los clics del usuario y decidirán qué pantalla mostrar o qué datos calcular.
  • 📂 wwwroot: La decoración. Ahí es donde guardaremos los archivos estáticos como imágenes, estilos CSS (colores, tamaños) y scripts de JavaScript.

Fase 2: Entity Framework, Contexto de Datos y Migraciones

En lugar de escribir código SQL manualmente para crear tablas, utilizamos un ORM llamado Entity Framework Core. Este sistema lee nuestras clases de C# y genera un guion (script) para construir la base de datos mediante comandos.

1. Entity Framework Core: El Traductor Universal

En el pasado, los programadores tenían que escribir largas y complejas consultas en lenguaje SQL puro para guardar o leer datos (ej. SELECT * FROM Funcionarios WHERE...). Hoy, utilizamos una tecnología llamada ORM (Object-Relational Mapper).

En .NET, nuestro ORM se llama Entity Framework Core. Funciona como un traductor bilingüe perfecto: tú le hablas en C# (usando modelos que se explicaron en la Fase 1) y él se encarga de traducirlo automáticamente a comandos de SQL Server por detrás. Esto nos ahorra horas de trabajo y previene muchísimos errores de seguridad.

    3. Migraciones: Construyendo la Base de Datos

    Tener el puente (ApplicationDbContext) y los planos (Modelos) no es suficiente. Tenemos que enviar a los obreros a construir las tablas reales en SQL Server. A este proceso se le llama Migraciones.

    Una migración es como tomarle una "fotografía" al estado actual de tu código C# y decirle a la base de datos que se actualice para verse exactamente igual a esa foto. Esto se hace en dos pasos a través de la consola:

    • Paso 1: Crear los planos de construcción

      El comando Add-Migration NombreDeLaMigracion (o dotnet ef migrations add) revisa qué cambió en tus modelos y genera un archivo con las instrucciones para la base de datos.

    • Paso 2: Ejecutar la construcción

      El comando Update-Database (o dotnet ef database update) toma esos planos y viaja hasta SQL Server para crear, borrar o modificar las tablas, columnas y relaciones físicamente.

    💡 Tip de Ingeniero: Si alguna vez agregas una nueva propiedad a tu modelo (por ejemplo, el Email de un funcionario) y olvidas correr estos dos comandos, el programa fallará porque el código buscará una columna en la base de datos que aún no existe. ¡Las migraciones mantienen todo sincronizado!

    Fase 3: Controladores (El Cerebro)

    El Controlador es el Mesero de nuestro restaurante. Recibe las peticiones del usuario (clics), procesa la información (hace cálculos matemáticos, valida reglas de negocio) y decide qué guardar en la base de datos.

    1. El Mesero en Acción

    Volviendo a nuestra analogía del restaurante, el Controlador es el Mesero. Su trabajo es fundamental y se divide en tres pasos exactos cada vez que un cliente hace clic en la página web:

    1. Escucha la petición: El usuario escribe una URL o hace clic en un botón (ej. "Guardar Reserva").
    2. Procesa los datos: El controlador va a la base de datos (nuestra cocina) usando el ApplicationDbContext para guardar o buscar la información necesaria.
    3. Sirve el resultado: Le entrega los datos listos a una Vista (el comedor) para que el usuario los vea en su pantalla.

    2. El Lenguaje de la Web: GET vs POST

    Para que el Mesero no se confunda, en el desarrollo web usamos dos "etiquetas" principales llamadas Verbos HTTP. Es crucial entender la diferencia antes de programar:

    • [HttpGet] "Tráeme información": Se usa cuando el usuario solo quiere VER algo. Por ejemplo, entrar a la página para ver la lista de empleados o abrir un formulario en blanco. Es como pedirle la carta al mesero.
    • [HttpPost] "Toma esta información": Se usa cuando el usuario LLENÓ un formulario y hace clic en "Guardar". Los datos viajan ocultos y seguros hacia el servidor. Es como entregarle la orden escrita al mesero para que la lleve a la cocina.

    Fase 4: Las Vistas y la Magia de Razor

    El HTML por sí solo es como un póster impreso: no sabe hacer matemáticas ni consultar bases de datos. Para inyectarle vida, usamos Razor. Razor nos permite usar el símbolo @ para escribir código C# directamente dentro del HTML.

    1. La Vista: El Comedor del Restaurante

    Siguiendo nuestra analogía, la Vista es el comedor limpio, decorado y la carta física que se le entrega al cliente. Es la capa visual con la que interactúan las personas. Un usuario común no entiende de bases de datos relacionales ni de inyección de dependencias en C#; él solo ve botones, formularios, tablas y colores.

    En ASP.NET Core MVC, las vistas no son simples archivos HTML estáticos independientes. Son archivos con la extensión .cshtml. Esto significa que combinan el poder estructural de HTML5 con la potencia lógica de C#.

    2. El Motor de Renderizado Razor

    Para lograr que HTML y C# convivan en un mismo archivo de forma fluida, Microsoft creó un motor de plantillas llamado Razor. Su funcionamiento es sumamente elegante: cada vez que el motor encuentra el símbolo de la arroba (@), sabe que lo que viene a continuación no es texto plano para el navegador, sino una instrucción lógica en C# que debe ejecutarse en el servidor antes de enviar la pantalla.

    Las directivas fundamentales de Razor que todo ingeniero debe dominar son:

    • @model (en minúscula): Define con qué tipo de "ingrediente" o molde de datos va a trabajar esta pantalla específica.
    • @foreach o @if: Permiten crear bucles dinámicos y condiciones lógicas directamente dentro del diseño visual.
    • Model (en mayúscula): Es la variable real que contiene la información que el controlador le envió a la mesa.

    3. Diseñando la Vista de Listado (Index.cshtml)

    Cuando el método Index() de nuestro controlador va a la base de datos y trae la lista de empleados, la envía a esta pantalla. Aquí usamos un ciclo para dibujar una fila de tabla por cada funcionario registrado automáticamente.

Estos atributos conectan las cajas de texto directamente con las variables de nuestro modelo, facilitando que el navegador capture la información de forma estructurada.

    Fase 5: Reportes y Exportación a Excel

    En el mundo real, los gerentes siempre pedirán reportes descargables. En .NET utilizamos una librería llamada ClosedXML para generar archivos Excel sin necesidad de tener Microsoft Office instalado en el servidor.

    1. La Petición de la Gerencia

    ¡Bienvenidos a la recta final! Hemos construido la cocina (Modelos), la despensa (Base de Datos), entrenado a los meseros (Controladores) y decorado el comedor (Vistas). Nuestro sistema ya permite ingresar, calcular y consultar datos de manera segura y automática.

    Sin embargo, en el mundo real, a los gerentes y administradores no siempre les basta con ver los datos en una pantalla web; su petición más común será: "Envíame ese reporte en Excel". Hoy aprenderemos a tomar la información viva de nuestra base de datos y convertirla, mediante código, en un archivo Excel (.xlsx) descargable, estructurado y con formato profesional.

    2. Instalando la herramienta correcta: ClosedXML

    En el pasado, para que C# creara un documento de Excel, necesitabas tener instalado Microsoft Office en el servidor, lo cual era ineficiente y costoso. Hoy en día usamos librerías que generan el archivo internamente manipulando el estándar OpenXML. La mejor y más amigable para este trabajo se llama ClosedXML.

    Paso a paso para instalarla:
    1. En el menú superior de Visual Studio, ve a Herramientas > Administrador de paquetes NuGet > Administrar paquetes NuGet para la solución...
    2. Ve a la pestaña Examinar (Browser).
    3. En la barra de búsqueda, escribe exactamente: ClosedXML.
    4. Selecciona el paquete cuyo autor es "ClosedXML", marca la casilla de tu proyecto (HotelAdmin) en el panel derecho y haz clic en Instalar.
    5. Si aparece un cuadro de diálogo pidiendo aceptar licencias o dependencias, haz clic en Aceptar.

    3. Explicación del "Bucle Mágico" para llenar el Excel

    Si recuerdan los ciclos for o while de sus primeras clases de fundamentos de programación, sabrán que sirven para repetir instrucciones rápidamente. Aquí usamos un ciclo foreach para recorrer la lista de reservas que sacamos de la base de datos e imprimirlas en el Excel.

    Imaginen una hoja de Excel como un plano cartesiano. Las coordenadas se leen como worksheet.Cell(Fila, Columna).

    • La variable de control: Antes de entrar al ciclo, creamos una variable int currentRow = 2; (Fila Actual). ¿Por qué empezamos en 2? Porque la fila 1 ya la ocupamos pintando los títulos (ID, Huésped, etc.).
    • La primera iteración: El foreach toma la primera reserva de nuestra lista de la base de datos. Usando la variable currentRow (que vale 2), empieza a escribir en la fila 2, columna 1 (r.Id), luego en la columna 2 (r.NombreHuesped), y así hasta la columna 8.
    • El incremento: Justo al final de las instrucciones de esa reserva, vemos la línea currentRow++;. Esto en C# significa "Súmale 1 al valor actual". Ahora currentRow vale 3.
    • La segunda iteración: El foreach toma la segunda reserva de la lista. Vuelve a ejecutar las mismas líneas de código, pero ahora, como la variable currentRow vale 3, escribirá todos los datos un renglón más abajo (en la Fila 3), evitando sobrescribir los datos de la primera reserva.
    • El final: Este proceso se repite en bucle tantas veces como reservas haya. Si hay 100 reservas, el código iterará y escribirá hasta la fila 101 de forma automática en apenas unos milisegundos.

    Simulador de Exportación

    Fase 7: Actualización de Datos (Edición)

    En cualquier sistema corporativo, los usuarios cometen errores y necesitan modificar registros. Aquí completamos la "U" del acrónimo CRUD (Update). Para editar, el Controlador usa un trabajo en equipo de dos métodos:

    • GET (Buscar y Mostrar): Va a la Base de Datos, busca la reserva y llena el formulario.
    • POST (Recibir y Actualizar): Recibe el formulario modificado y sobrescribe la Base de Datos.

    1. El Desafío de las Llaves Foráneas (Relaciones 1 a Muchos)

    En la base de datos, una reserva no puede flotar sola en el espacio; por reglas del negocio, obligatoriamente debe estar asociada al funcionario que realizó la venta (Llave Foránea: FuncionarioId). Cuando un usuario va a registrar una reserva, no podemos pedirle que escriba el número de ID numérico del empleado de memoria, ya que eso causaría errores fatales si digita un número que no existe.

    La solución elegante en la ingeniería de software es implementar un Menú Desplegable Dinámico (Select). El controlador irá a la tabla de Funcionarios, extraerá los nombres completos y sus IDs, y los empaquetará en una lista de selección que viajará a la Vista.

            if (dias <= 0) dias = 1;

            decimal tarifa = reserva.TipoHabitacion switch

        {
            if (id != reserva.Id) return NotFound();

            ModelState.Remove("Funcionario"); 

            if (ModelState.IsValid)
            {
                try
                {
                    int dias = (reserva.CheckOut - reserva.CheckIn).Days;
                    if (dias <= 0) dias = 1;

      El archivo Program.cs es el motor que arranca todo nuestro sistema. Aquí es donde inyectamos la conexión a la base de datos, configuramos las reglas de seguridad (Login) y definimos qué pantalla se mostrará por defecto al abrir la aplicación.

      Ruta del archivo: HotelAdmin/Program.cs 
      using HotelAdmin.Data;
using HotelAdmin.Models;
using Microsoft.AspNetCore.Identity;
using Microsoft.EntityFrameworkCore;

var builder = WebApplication.CreateBuilder(args);

var connectionString = builder.Configuration.GetConnectionString("DefaultConnection")
    ?? throw new InvalidOperationException("Connection string 'DefaultConnection' not found.");

builder.Services.AddDbContext<ApplicationDbContext>(options =>
    options.UseSqlServer(connectionString));

builder.Services.AddIdentity<ApplicationUser, IdentityRole>(options => {
    options.Password.RequireDigit = false;
    options.Password.RequiredLength = 6;
    options.Password.RequireNonAlphanumeric = false;
    options.Password.RequireUppercase = false;
    options.Password.RequireLowercase = false;
})
.AddEntityFrameworkStores<ApplicationDbContext>()
.AddDefaultTokenProviders();

builder.Services.ConfigureApplicationCookie(options =>
{
    options.LoginPath = "/Account/Login";
    options.LogoutPath = "/Account/Logout";
    options.AccessDeniedPath = "/Account/AccessDenied";
});

builder.Services.AddControllersWithViews();

var app = builder.Build();

if (!app.Environment.IsDevelopment())
{
    app.UseExceptionHandler("/Home/Error");
    app.UseHsts();
}

app.UseHttpsRedirection();
app.UseStaticFiles();

app.UseRouting();

app.UseAuthentication();
app.UseAuthorization();

app.MapControllerRoute(
    name: "default",
    pattern: "{controller=Dashboard}/{action=Index}/{id?}");

app.Run();
